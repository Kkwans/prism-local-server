# 任务 10.2 手动性能测试指南

## 测试概述

本指南提供详细的手动性能测试步骤，用于验证 Prism Local Server Tauri 版本是否满足性能需求。

**测试目标**:
- 冷启动时间 ≤ 1.5 秒
- 空闲状态内存占用 ≤ 40MB
- 运行 5 个服务时内存占用 ≤ 150MB
- 小文件响应时间 ≤ 10ms
- 大视频首帧加载时间 ≤ 100ms
- 打包体积 ≤ 15MB

---

## 测试前准备

### 1. 构建 Release 版本

如果还没有构建 Release 版本，请执行：

**PowerShell:**
```powershell
cd prism-local-server-tauri
npm run tauri:build
```

**CMD:**
```cmd
cd prism-local-server-tauri
npm run tauri:build
```

构建完成后，EXE 文件将自动复制到项目根目录：`prism-local-server-v3.0.0.exe`

### 2. 准备测试工具

- **秒表或手机计时器**: 用于测量冷启动时间
- **任务管理器**: 用于测量内存占用
- **浏览器开发者工具**: 用于测量 HTTP 响应时间
- **文件资源管理器**: 用于查看 EXE 文件体积

### 3. 准备测试资源

确保 `tests/test_resources/` 目录包含以下文件：
- `index.html` - 简单的 HTML 页面
- `css/style.css` - CSS 样式文件
- `js/script.js` - JavaScript 文件
- `images/test-image.png` - 图片文件（约 500KB）
- `videos/test-video.mp4` - 视频文件（约 50MB）

### 测试 4: 小文件响应时间（目标 ≤ 10ms）

1. **启动服务**

2. **打开浏览器开发者工具**

## 性能优化建议

### 冷启动时间优化

如果冷启动时间超过 1.5 秒，可以尝试：

- 减少应用启动时的初始化操作
- 延迟加载非关键模块
- 优化 Rust 依赖项编译
- 使用 `lazy_static` 延迟初始化全局变量
- 检查是否有阻塞的同步操作

### 内存占用优化

如果内存占用超标，可以尝试：

- 使用内存分析工具检查是否有内存泄漏
- 优化 Tokio 运行时配置
- 减少不必要的数据缓存
- 使用 `Arc` 和 `Weak` 引用避免循环引用
- 及时释放不再使用的资源

### 响应时间优化

如果响应时间超标，可以尝试：

- 优化静态文件处理器
- 使用零拷贝技术（`tokio::io::copy`）
- 启用 `TCP_NODELAY` 选项
- 优化文件读取缓冲区大小
- 使用异步文件操作

### 打包体积优化

如果打包体积超标，可以尝试：

- 进一步优化 `Cargo.toml` 的 release profile
- 移除未使用的依赖项
- 使用 `strip` 移除调试符号
- 考虑使用 UPX 压缩 EXE（可选）
- 检查是否包含了不必要的资源文件

## 附录：常见问题

### Q1: 如何准确测量冷启动时间？

**A**: 建议使用以下方法之一：
1. 使用录屏工具（如 OBS）录制启动过程，然后逐帧分析
2. 使用 PowerShell 脚本自动测量（见 `performance_test.ps1`）
3. 使用秒表，从双击 EXE 到窗口完全显示

### Q2: 任务管理器中看不到进程怎么办？

**A**: 
1. 确保应用已经启动
2. 在任务管理器中切换到"详细信息"标签
3. 按名称排序，查找 `prism-local-server-tauri.exe`
4. 如果还是找不到，检查应用是否启动失败

### Q3: 如何测量视频的 Range Request？

**A**:
1. 在浏览器开发者工具的 Network 标签中
2. 找到视频文件的请求
3. 点击请求，查看"Headers"（请求头）
4. 查找 `Range: bytes=xxx-xxx` 请求头
5. 查看响应状态码是否为 `206 Partial Content`

### Q4: 性能测试应该在什么环境下进行？

**A**:
- 建议在干净的 Windows 11 环境中测试
- 关闭其他占用资源的程序
- 确保系统资源充足（至少 8GB RAM，4 核 CPU）
- 使用稳定的网络环境

### Q5: 如果多个指标都未达标怎么办？

**A**:
1. 首先检查是否是测试环境的问题
2. 检查 Release 版本是否正确构建
3. 查看 Rust 日志，寻找性能瓶颈
4. 使用性能分析工具（如 `perf`, `flamegraph`）
5. 逐个优化未达标的指标
